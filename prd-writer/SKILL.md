# PRD Writer

Transform rough product ideas into structured, implementation-ready PRDs optimized for Claude Code handoff.

## Quick Start

This skill creates PRDs through a conversational process:

1. **Receive idea** — User shares rough concept
2. **Clarify gaps** — Ask targeted questions for missing context
3. **Draft PRD** — Create structured document with all sections
4. **Generate output** — Produce both .docx and .md files

## Core Philosophy

### What Makes a Great PRD for AI Coding

PRDs for AI coding assistants differ from traditional enterprise PRDs:

- **Explicit over implicit** — Describe exact behaviors, not vague goals
- **Testable over aspirational** — Every feature needs clear pass/fail criteria
- **Scoped over comprehensive** — Define what NOT to build as clearly as what to build
- **Structured over narrative** — Use consistent patterns AI can parse

### The "What" Not the "How"

Following Marty Cagan's guidance: PRDs explain WHAT to build and WHY, not HOW to build it. Technical implementation details (specific code patterns, architecture decisions) belong in the Implementation Notes section but should allow Claude Code flexibility to propose solutions.

### Multi-Agent by Default

PRDs generated by this skill are structured for **multi-agent orchestration** in Claude Code:

- **Work is parallelizable** — Features split into frontend, backend, database, and test tasks
- **Clear boundaries** — Each agent task has explicit inputs, outputs, and dependencies
- **Test-driven validation** — A dedicated testing agent writes tests from acceptance criteria FIRST
- **Orchestrator pattern** — Main agent coordinates, delegates, and validates

See `references/multi_agent_orchestration.md` for detailed patterns.

## Conversation Workflow

### Step 1: Receive Initial Idea

Accept the user's rough idea in any form. Could be:
- A single sentence pitch
- A comparison ("like Airbnb but for X")
- A problem statement
- Feature request for existing product

### Step 2: Identify Missing Information

Analyze the idea against required PRD sections. Ask clarifying questions ONLY for genuinely missing context. Group questions conversationally (2-3 max per message).

**Required context to clarify:**

| Category | Key Questions |
|----------|---------------|
| Problem | What specific problem does this solve? Who experiences it? How painful is it? |
| User | Who is the primary user? Be specific (not "everyone"). What's their context? |
| Solution | What's the core value proposition? What makes this different from alternatives? |
| Scope | Is this an MVP, feature, or full product? What's explicitly out of scope? |
| Success | How will you know this works? What metrics matter? |
| Technical | Any existing system to integrate with? Tech preferences? Constraints? |

**Smart question selection:**

- If the user provides a detailed problem statement, skip problem questions
- If they mention "MVP" or "POC", assume minimal scope
- If they compare to existing products, use that as baseline
- If building for an existing system, ask about integration points

### Step 3: Draft the PRD

Once context is sufficient, generate the complete PRD using the structure in the PRD Template section below.

### Step 4: Generate Outputs

Create both formats:
1. **Markdown (.md)** — For Claude Code consumption, version control
2. **Word Document (.docx)** — For stakeholder sharing, formal documentation

Use the docx skill for .docx creation.

         │         ├── Task 2.2: TEST AGENT

## Writing Guidelines

### Acceptance Criteria Format

Use Given/When/Then (Gherkin) format for testable criteria:

```
Given [precondition/context]
When [action taken]
Then [expected result]
```

**Examples:**

```
Given a user is logged in and on the dashboard
When they click "Create Invoice"
Then a new invoice form appears with today's date pre-filled

Given an invoice is overdue by 7 days
When the system runs the daily reminder job
Then an email is sent to the client with invoice details
```

**Best practices:**
- Each criterion should be independently testable
- 1-3 acceptance criteria per feature (if more, split the feature)
- Cover the happy path first, then edge cases
- Include error states

### User Flow Numbering

Always number steps sequentially. Each step should be one discrete action.

**Good:**
```
1. User clicks "Sign Up"
2. System displays registration form
3. User enters email and password
4. User clicks "Create Account"
5. System validates input
6. System creates account and redirects to dashboard
```

**Bad:**
```
- User signs up by clicking the button and filling out the form
- System processes everything and shows the dashboard
```

### Feature Behavior Pattern

Every feature needs the Trigger → Action → Result pattern:

| Element | Question | Example |
|---------|----------|---------|
| Trigger | What starts this? | "User clicks Add to Cart" |
| Action | What does the system do? | "System adds item to cart, updates count" |
| Result | What does user see? | "Cart icon shows updated count, toast confirms add" |

### Scope Language

Be explicit with scope boundaries:

- ✅ "NOT building payment processing — users pay externally"
- ✅ "NOT building mobile app — web only for MVP"
- ❌ "We might add this later"
- ❌ "Out of scope for now"

## Technical Defaults

When users don't specify technical preferences, recommend this POC-friendly stack:

### Database & Auth
- **Supabase** — PostgreSQL + Auth + Real-time in one service
- Free tier sufficient for POC testing
- Built-in row-level security

### Frontend
- **React** or **Next.js** — Wide ecosystem, Claude Code proficiency
- **Tailwind CSS** — Utility-first, consistent styling

### Deployment
- **Vercel** (Next.js) or **Netlify** (React) — Free tier, simple CI/CD

### Testing (Required for Multi-Agent)
- **Vitest** — Fast unit/integration tests, Jest-compatible
- **Playwright** — E2E testing, cross-browser
- **Testing Library** — Component testing utilities

### Always Ask About:
- Existing codebases or repos to integrate with
- Authentication requirements (social login? email only?)
- Data sensitivity / compliance needs
- Expected user volume

## Quality Checklist

Before delivering the PRD, verify:

### Completeness
- [ ] One-sentence description is specific and clear
- [ ] Problem statement includes evidence (quantitative + qualitative)
- [ ] Target user is defined with specific attributes
- [ ] Each feature has trigger, action, and result
- [ ] User flows are numbered step-by-step
- [ ] "Out of Scope" section exists and is specific
- [ ] Acceptance criteria are testable (Given/When/Then)
- [ ] Success metrics have measurable targets

### Claude Code Readiness
- [ ] Database schema provided (if data storage needed)
- [ ] API endpoints defined (if backend needed)
- [ ] Tech stack recommendations included
- [ ] No ambiguous requirements ("should be user-friendly")
- [ ] Error states documented for each feature

### Multi-Agent Readiness
- [ ] Section 10 (Multi-Agent Task Breakdown) is complete
- [ ] Each feature mapped to agent responsibilities (DB, Backend, Frontend, Test)
- [ ] AC-to-Test mapping table populated
- [ ] Verification commands specified for each agent
- [ ] Dependencies between tasks clearly marked
- [ ] Phase boundaries defined (what must complete before next phase)

### Test Coverage
- [ ] Every acceptance criterion has a corresponding test entry
- [ ] E2E tests defined for each user flow
- [ ] Error cases have explicit test requirements
- [ ] Coverage target specified (default: 80%)

### Document Quality
- [ ] Consistent formatting throughout
- [ ] No placeholder text remaining
- [ ] All tables properly formatted
- [ ] Links and references included where helpful

## Output Generation

### Markdown Output

Save the PRD as `PRD-[ProductName]-v[X].md` in the outputs directory.

### Word Document Output

Use the docx skill to create a professional .docx version:

1. Follow docx skill guidelines for formatting
2. Include table of contents
3. Use proper heading hierarchy (H1 for title, H2 for sections)
4. Apply consistent styling
5. Save as `PRD-[ProductName]-v[X].docx`

### File Naming Convention

```
PRD-[ProductName]-v1.md
PRD-[ProductName]-v1.docx
```

## Example Conversation Flow

**User**: "I want to build an app for tracking unpaid invoices and sending reminders"

**Claude**: That's a useful tool for freelancers and small businesses. Let me ask a few questions to make sure the PRD captures what you need:

1. **Who's the primary user?** Freelancers? Small business owners? Accounting teams?
2. **What's the current pain?** Are they forgetting to follow up? Spending too much time tracking?
3. **MVP scope**: For the first version, should this just track invoices and send basic email reminders, or do you need features like payment processing, recurring invoices, or client portals?

**User**: "Freelance designers who send 5-10 invoices a month. They forget to follow up and lose money. Just tracking and reminders for MVP."

**Claude**: Got it. Two more quick questions:

1. **Reminder timing**: Should reminders go out automatically (e.g., 7 days after due date) or should users trigger them manually?
2. **Tech preferences**: Any existing tools this needs to integrate with, or should I recommend a standard stack (Supabase + Next.js)?

**User**: "Automatic reminders at 7 days overdue. No existing tools, recommend whatever's easiest to build."

**Claude**: Perfect. I have everything needed. Let me generate the PRD for Invoice Reminder.
